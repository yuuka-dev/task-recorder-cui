# 英語対応 (i18n) 設計書 (Phase 2.1)

- 作成日: 2026-04-15
- 対象バージョン: v1.1.0 (予定)
- 対象プロジェクト: task-recorder-cui
- 関連 spec: [2026-04-15-timer-design.md](2026-04-15-timer-design.md) (並列実装)

## 1. スコープと位置づけ

### やること
- すべての UI メッセージ (メニュー、サブコマンド出力、エラー文、プロンプト) を日本語/英語で切替可能にする
- 言語切替の優先順位: CLI フラグ > 設定ファイル > 環境変数 > デフォルト (ja)
- `locales/ja.py` / `locales/en.py` の 2 モジュールでメッセージを管理する
- README.en.md を追加する

### やらないこと
- **カテゴリの多言語化** (`display_name` はユーザ入力なので触らない。英語モードでも「ゲーム」は「ゲーム」のまま)
- 複数形処理 (英語の単数/複数は単純な文字列で十分な範囲に留める、例: `1 minute` / `2 minutes` は分岐なしで `1m` / `2m` に揃える)
- 翻訳ツール連携 (.po / .mo / gettext / crowdin 等)
- 日付・時刻フォーマットのロケール別化 (`2026-04-15 (Tue)` 形式は両言語で共通)
- カテゴリ seed データの英語版 (初期 3 カテゴリ `ゲーム/学習/開発` は日本語固定、ユーザが必要なら `tsk cat rename game "Game"` で変更する)

### CLAUDE.md の改定点
本 spec 確定と同じコミット (または直後のコミット) で、`CLAUDE.md` を以下のように更新する:

- 「Phase 2 計画」節の「英語対応 (国際化)」項目を「Phase 2.1 実装済み」に移動
- コーディング規約に「ユーザ向け文字列は必ず `i18n.t()` 経由」を追記

## 2. アーキテクチャ

### 2.1 メッセージカタログ方式

採用: **dict ベース自前実装 (②)**。

- 却下: gettext (③, `.mo` コンパイル必要、CI 複雑化), fluent (④, オーバースペック)
- 理由: メッセージ数が数十個レベル、標準ライブラリだけで完結、テストしやすい

### 2.2 メッセージキー命名

採用: **フラット定数 (B)**。

```python
# src/task_recorder_cui/locales/ja.py
SESSION_ACTIVE = "現在: [{category}] {description} ({elapsed}経過)"
SESSION_NONE = "現在: 記録なし"
TIMER_FIRED = "タイマー経過しました"
TIMER_PROMPT = "タイマー (例: 2h30m、空欄でスキップ): "
ERROR_INVALID_KEY = "カテゴリ key が不正です: {key}"
# ...

# src/task_recorder_cui/locales/en.py
SESSION_ACTIVE = "Active: [{category}] {description} ({elapsed})"
SESSION_NONE = "No active session"
TIMER_FIRED = "Timer expired"
TIMER_PROMPT = "Timer (e.g. 2h30m, blank to skip): "
ERROR_INVALID_KEY = "Invalid category key: {key}"
# ...
```

- 定数名は `UPPER_SNAKE_CASE`、アルファベット順 or グループ順 (SESSION_*, TIMER_*, ERROR_*)
- 利点: IDE 補完、`from locales import ja as M; M.SESSION_ACTIVE` でタイポを import 時に検出できる
- 欠点: キー追加時に両言語ファイルに書き足す必要がある → CI で片方欠けを検出 (後述 §6)

### 2.3 レイヤ

```
UI 層 (cli.py / menu.py / commands/*.py)
    ↓ i18n.t("SESSION_ACTIVE", category=..., description=..., elapsed=...)
i18n.py (新設 — t() ヘルパ、言語選択、フォールバック)
    ↓
locales/ja.py, locales/en.py (モジュール直参照)
```

- `t(key: str, **fmt) -> str` が唯一の公開 API
- `io.py` の `print_line` / `print_warning` / `print_error` は **変更不要** (呼び出し側で `t()` してから渡す)

### 2.4 `t()` の実装

```python
# src/task_recorder_cui/i18n.py
import os
from typing import Any

from task_recorder_cui.locales import en, ja

_LOCALES = {"ja": ja, "en": en}
_current_lang: str | None = None

def set_lang(lang: str | None) -> None:
    """明示的な言語指定 (CLI フラグ or 設定から)。None で自動判定に戻る。"""
    global _current_lang
    _current_lang = lang

def current_lang() -> str:
    """現在の有効な言語を返す。優先順位: set_lang > config.ui.lang > LC_ALL/LANG env > "ja"。

    未対応言語 (fr など) は "ja" にフォールバックする (警告なし)。
    """
    if _current_lang is not None:
        return _current_lang
    from task_recorder_cui.config import load_config  # 遅延 import で循環回避
    cfg_lang = load_config().ui.lang
    if cfg_lang in _LOCALES:
        return cfg_lang
    for env_var in ("LC_ALL", "LANG"):
        env = os.environ.get(env_var, "")
        prefix = env.split("_", 1)[0].lower()
        if prefix in _LOCALES:
            return prefix
    return "ja"

def t(key: str, **fmt: Any) -> str:
    """メッセージキーを現在言語で解決し、{placeholder} を埋める。

    言語モジュールに該当キーが無ければ ja にフォールバック、ja にも無ければ
    key 自体を返す (開発時のタイポを可視化)。
    """
    lang = current_lang()
    mod = _LOCALES.get(lang, ja)
    template = getattr(mod, key, None)
    if template is None and mod is not ja:
        template = getattr(ja, key, None)
    if template is None:
        return key
    return template.format(**fmt)
```

## 3. 言語選択

### 3.1 優先順位

| 順位 | ソース | 例 |
|---|---|---|
| 1 | CLI フラグ `--lang` | `tsk --lang en week` |
| 2 | 設定ファイル `ui.lang` | `~/.config/tsk/config.toml` |
| 3 | 環境変数 `LANG` / `LC_ALL` | `LANG=en_US.UTF-8 tsk now` → `en` |
| 4 | デフォルト | `ja` |

### 3.2 `--lang` フラグ

- すべてのサブコマンドでグローバル利用可
- `argparse` のグローバル引数として `cli.py` に定義
- 値のバリデーション: `ja` / `en` のみ許容、それ以外はエラー

### 3.3 環境変数の解釈

- `LC_ALL` > `LANG` の順
- 値の先頭 `en` / `ja` を見る (`en_US.UTF-8` → `en`, `ja_JP.UTF-8` → `ja`)
- 未対応言語 (`fr` 等) は `ja` にフォールバック、警告は出さない

## 4. 対応するメッセージの洗い出し

実装時に以下のファイルを順に grep して網羅する:

- `src/task_recorder_cui/cli.py` — `--help` 以外の print
- `src/task_recorder_cui/menu.py` — メニュー全行、プロンプト
- `src/task_recorder_cui/commands/*.py` — 全コマンドの出力・エラー
- `src/task_recorder_cui/io.py` — ラッパなので対象外 (中身の呼び出し側が `t()` 経由)
- `src/task_recorder_cui/utils/validate.py` — バリデーションエラー文

**argparse の `--help` 文字列**: 今回は **英語固定とする**。理由は argparse の国際化に標準ライブラリの `gettext` 機構を組み込む必要があり、本 spec の dict 方式と相性が悪いため。`ArgumentParser(description=..., help=...)` に渡す文字列はすべて英語で書く。日本語 `--help` 対応は Phase 2.2 以降の課題。

## 5. 設定ファイル連携

設定ファイル `~/.config/tsk/config.toml` の `ui.lang` を参照する。設定ファイル実装は **timer-design.md の §4** が先行して担当するため、本 spec はそのスキーマを利用するだけ:

```toml
[ui]
lang = "ja"  # ja / en、空文字 or 未設定なら LANG 環境変数から推定
```

実装順: `config.py` が先 (timer 側)、その後 `i18n.py` が `config.py` から読み取る。

## 6. テスト戦略

| 対象 | テスト種別 | 備考 |
|---|---|---|
| `t()` のキー解決 | ユニット | 正常 / フォールバック (en に無ければ ja) / 完全未定義 (key をそのまま返す) |
| `current_lang()` の優先順位 | ユニット | set_lang / config / LANG env / デフォルト の 4 分岐 |
| メッセージキーの網羅性 | 静的チェック | `ja.py` と `en.py` のパブリック定数集合が一致するか (pytest で実装) |
| 各コマンドが `t()` 経由で出力しているか | 手動レビュー or grep | `print\(` / `print_line\(".+"\)` の直書き検知 |

### 6.1 キー網羅テスト例

```python
# tests/test_i18n_coverage.py
import inspect
from task_recorder_cui.locales import ja, en

def _public_constants(mod) -> set[str]:
    return {name for name, val in inspect.getmembers(mod)
            if name.isupper() and isinstance(val, str)}

def test_ja_en_have_same_keys():
    assert _public_constants(ja) == _public_constants(en)
```

このテストで片方追加し忘れを CI が落とす。

## 7. フォーマッタ整合

- `ja` では全角スペース混在が読みやすい文もあるが、**半角スペースで揃える** (grep しやすさ、将来的な等幅表示)
- `en` メッセージは文頭大文字、句点なしを原則 (例: `No active session` ではなく `No active session.` にしない — CLI 慣習に合わせる)
- 数値・時間フォーマット (`2h30m`, `53%` 等) は両言語で共通

## 9. 依存ライブラリとテスト

### 9.1 追加依存

- なし (標準ライブラリのみ)

### 9.2 既存テストへの影響

- 既存テストで日本語メッセージを assert しているものは `t()` 経由に書き換え、`en` ロケールでの期待値も追加
- 具体例:
  ```python
  # 変更前
  assert "現在: 記録なし" in output
  # 変更後
  from task_recorder_cui.i18n import t, set_lang
  set_lang("ja")
  assert t("SESSION_NONE") in output
  ```

## 10. 実装順序 (writing-plans 向けガイド)

1. `locales/__init__.py`, `locales/ja.py`, `locales/en.py` を作成、既存文字列を抽出してキー化
2. `i18n.py` 新設、`t()` と `current_lang()` 実装
3. `cli.py` に `--lang` グローバルフラグ追加
4. `menu.py` / `commands/*.py` の文字列を `t()` 呼び出しに置換 (ファイル単位で段階的に)
5. 既存テストを `t()` ベースに書き換え、`en` ケース追加
6. キー網羅テスト (`test_i18n_coverage.py`) を追加
7. `README.en.md` 作成、相互リンク追加
8. `CLAUDE.md` 更新

## 11. タイマー機能との相互依存

- **config.py の先行実装**: timer-design.md §4 で設定ファイル基盤 (`config.py`) を作る。i18n ブランチの実装はこれを import する
- **メッセージキー追加ルール**: timer 側で新規追加する文字列 (`TIMER_PROMPT`, `TIMER_FIRED`, `TIMER_INVALID_FORMAT` 等) は **タイマー実装側の worktree で** `locales/ja.py` と `locales/en.py` の両方に追加する。i18n 側は既存文字列の置換に集中する
- **マージ戦略**: 両 worktree が `locales/*.py` を触るため、`locales/*.py` でコンフリクトが起きやすい。推奨する運用:
  1. i18n ブランチを先に main にマージ (基盤 + 既存文字列のキー化)
  2. timer ブランチを main から rebase で最新取り込み
  3. timer ブランチでタイマー固有のキーを追加 → main にマージ
- **マージ順の入れ替え**: どうしても timer 先にしたい場合は、i18n ブランチが rebase で追従する。`locales/*.py` のコンフリクトは「両方のキーを残す」方針で解決
